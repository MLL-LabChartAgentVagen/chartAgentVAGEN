## 1.2 数据流 / 交接关系

- Phase 1 场景上下文 → LLM 代码生成提示词 → LLM → Python SDK 脚本

- `build_fact_table()` → `FactTableSimulator`（列声明 → 关系声明）
- `FactTableSimulator.generate()` pipeline：α（骨架 / 非度量列） → β（按拓扑顺序生成度量） → γ（模式注入） → δ?（可选真实感注入） → `_post_process()` → `(DataFrame, schema_metadata)`
- α 内部流程：独立根节点 → 跨组依赖根节点 → 组内子节点 → 时间派生列（均按拓扑顺序）
- β 内部流程：随机根节点度量 → 结构性度量（按度量子 DAG 拓扑顺序）

- Schema 元数据 → Phase 3（视图提取、QA 生成）

## 1.3 设计决策 / 约束条件

- **代码即 DGP（Code-as-DGP）**：LLM 编写可执行的 Python 代码，调用类型安全的 SDK，替代基于 JSON 的 DGP 配置
- **闭合式声明（Closed-form declaration）**：每个度量声明一次，作为完整的数据生成程序，不做增量式补丁
- **所有列声明（第 1 步）必须先于所有关系声明（第 2 步）**
- **原子粒度（Atomic grain）**：每行 = 一个不可分割的事件，不进行笛卡尔积展开
- **事件级生成（Event-level generation）**：行作为独立原子事件生成，而非通过多维展开
- **度量 DAG 约束**：所有度量依赖必须无环；结构性度量只能引用已声明的度量；生成顺序由拓扑排序决定
- **仅根节点可参与跨组依赖**：`add_group_dependency()` 仅限组根节点（无父节点）；根节点级依赖图必须无环
- **正交性在组级别传播**：声明两个组正交，使所有跨组列对相互独立
- **跨组独立性是显式声明的，不是默认行为**
- **每个分类列恰好属于一个维度组**
- **时间列是一类特殊的维度组**，其派生列自动提取
- **支持的分布族**：gaussian、lognormal、gamma、beta、uniform、poisson、exponential、mixture
- **模式类型**：outlier_entity、trend_break、ranking_reversal、dominance_shift、convergence、seasonal_anomaly
- **确定性引擎**：给定 `(declarations, seed)`，`generate()` 完全确定 — 不调用 LLM
- **三层验证均不调用 LLM**：L1 结构层、L2 统计层、L3 模式层
- **自动修复循环**：验证失败触发参数调整和重新执行，而非重新调用 LLM（最多 3 次重试）
- **执行错误反馈循环**：SDK 抛出类型化异常；代码 + traceback 反馈给 LLM 修复（最多 3 次重试）
- **LLM 输出硬性约束**：≥2 个维度组（每组 ≥1 个分类列）、≥2 个度量、≥1 个正交声明、≥1 个结构性度量、≥2 个模式注入、纯合法 Python、无环 DAG、每个符号效应必须有明确的数值定义
- **权重自动归一化；拒绝空值；验证父节点存在性**
- **`param_model` 采用截距 + 效应求和模型**：θⱼ = β₀ + Σ βₘ(Xₘ)
- **Schema 元数据是 Phase 2 与 Phase 3 之间的契约**
- **在相同 seed 下保证逐比特可复现**

# Phase 2 — 架构图谱

## 2a. 一段通俗摘要

本系统是一个合成表格数据生成框架。其中，LLM 编写一段 Python 脚本，调用类型安全的 SDK（`FactTableSimulator`），以代码而非 JSON 配置的形式声明列、分布、列间依赖和统计模式。SDK 内部会基于所有声明（分类层级、时间派生、随机度量、结构性度量）构建统一的有向无环图，随后由确定性引擎按拓扑顺序遍历该 DAG，将每行作为独立原子事件生成：先按根节点 → 层级 → 跨组依赖的顺序采样分类列和时间列，再从参数受分类上下文约束的参数化分布中采样随机度量，接着通过引用上游度量的公式计算结构性度量，最后注入叙事驱动的异常模式以及可选的数据缺陷。生成的 DataFrame 与结构化 Schema 元数据字典将经过三层确定性验证器（结构完整性、统计保真度、模式存在性）的检验，并配有自动修复重试循环。若 LLM 的原始脚本存在执行错误，则由独立的反馈循环将类型化异常发回 LLM 进行针对性修复。Schema 元数据作为下游阶段（视图提取、QA 生成）的数据契约向后传递。

# Phase 3 — 核心概念深度解析

## 风险排序

1. **完整生成 DAG vs. 度量子 DAG** — 两个相互重叠的 DAG 控制生成顺序；混淆二者会导致拓扑排序错误或遗漏依赖。
2. **随机度量 vs. 结构性度量的区别** — 边界决定了列是采样还是计算得出；理解错误会产生统计属性不符的值，并导致验证失败。
3. **维度组与仅根节点可参与跨组的约束** — 组抽象控制合法依赖的范围；对"根节点"的误解会导致非法依赖声明，本应抛出异常。
4. **截距 + 效应求和的 `param_model`** — 参数计算模型是条件分布规范的核心；实现错误会静默地产生错误分布。
5. **两个独立的重试循环（执行错误 vs. 验证自动修复）** — 两个循环的触发条件、修复策略和作用范围各不相同；混淆二者会导致控制流紊乱。

---

### 概念 1：完整生成 DAG vs. 度量子 DAG

**是什么：** 系统维护两个 DAG 视图。*完整生成 DAG* 包含所有列（分类根节点、经 `parent` 指针连接的分类子节点、时间派生列、经 `add_group_dependency` 连接的跨组依赖、随机度量、结构性度量）。*度量子 DAG* 是其子集，仅包含度量列及其相互依赖边（结构性 → 随机的引用关系）。两者均通过拓扑排序求解，但完整 DAG 决定所有列的整体生成顺序，而度量子 DAG 仅决定步骤 β 内部的排序。

**为何存在：** 分类列和时间派生列必须在任何度量将其用作预测变量之前就已存在。同时，结构性度量必须等待其上游度量生成完毕。单一统一的 DAG 同时捕获这两种依赖，确保生成列 X 时，X 所依赖的所有列均已具有值。

**具体类比：** 把它类比为 Make 这样的构建系统。完整 DAG 是所有构建目标（库、目标文件、可执行文件）的完整依赖图。度量子 DAG 类似于只看可执行文件及其相互之间的链接时依赖 —— 这是一个有用的子视图，但编译时仍需要完整图来决定何时构建每个 `.o` 文件。

**常见误解：** 开发者可能认为度量 DAG 是*唯一的* DAG，从而按声明顺序处理分类列，而非按拓扑顺序。当某个分类列存在对另一个分类列的跨组依赖时（例如 `payment_method` 依赖 `severity`），这会出错：若 `severity` 尚未生成，则无法对 `payment_method` 进行条件采样。

**Python 实现提示：** 使用一个 `networkx.DiGraph`，基于所有列声明和依赖边构建，调用一次 `nx.topological_sort()`。生成时，遍历这一单一顺序，根据列类型（分类 / 时间 / 随机 / 结构性）分发到对应的采样函数。

---

### 概念 2：随机度量 vs. 结构性度量的区别

**是什么：** *随机根节点度量* 直接从具名分布中采样，其参数仅是分类上下文的函数 —— 在 DAG 中不存在来自其他度量的入边。*结构性（派生型）度量* 通过公式确定性地计算得出，该公式引用其他度量以及分类效应，并可附加可选的加性噪声。两者的区分是二元的：若某度量的值依赖任何其他度量列，则为结构性度量；否则为随机度量。

**为何存在：** 旧设计允许增量式补丁（先添加分布，再添加依赖，再添加相关性），导致声明分布与实际分布产生偏差。通过让每个度量自成一体，系统拥有单一的事实来源：随机度量可用 KS 检验对照其声明分布进行验证，而结构性度量则通过检验残差（实际值 − 公式预测值）是否符合声明的噪声分布来验证。

**具体类比：** 在电子表格中，随机度量就像含 `=RANDBETWEEN(...)` 的单元格 —— 其值仅来自随机生成器和一组参数。结构性度量则像含 `=A1 * 12 + B1 + RANDBETWEEN(...)` 的单元格 —— 是引用其他单元格的公式，再加上一些抖动。

**常见误解：** 认为两种类型都"受分类上下文影响"，因此内部运作方式相同。实际上并非如此：随机度量使用 `param_model`（截距 + 效应 → 分布参数 → 采样），而结构性度量使用 `formula` 字符串 + `effects` 字典 + `noise` 规范（求解表达式 → 加分类附加效应 → 加噪声）。对结构性度量使用 `param_model`，或对随机度量使用 `formula`，均属类型错误。
